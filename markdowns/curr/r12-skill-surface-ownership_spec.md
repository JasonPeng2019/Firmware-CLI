# R12 Skill Surface Ownership And Sync Vocabulary

> STATUS: ACTIVE DOC CLEANUP - vocabulary layer for R12 provider-native skill
> projection, model-native fallback loading, and Wave 1.6-B native skill sync.

## Goal

FirmCLI docs must distinguish three surfaces that were previously described with
overlapping `.codex`, `.claude`, "client-owned", "runtime", and "native" wording.

The product rule is:

> The provider can see projected preloaded skills and user-owned native skill
> folders. The provider may only natively edit user-owned skill folders. Actual
> skills to preload stay FirmCLI/client-owned and are synchronized or projected
> by the brain, not edited in place by the provider.

## The Three Skill Surfaces

### 1. Projected preloaded skills

Projected preloaded skills are generated, provider-visible runtime copies/views
of FirmCLI-owned skills.

Examples:

```text
<provider-runtime>/.codex/skills/<skill_id>/
<provider-runtime>/.claude/skills/<skill_id>/
```

Properties:

- Generated by FirmCLI before or during a provider run.
- Derived from actual FirmCLI skill source packages or a validated registry.
- Visible to CLI providers because Codex/Claude native skill discovery requires
  provider-shaped folders.
- Disposable and regenerable.
- Marked with FirmCLI projection provenance such as
  `.firmcli-provider-native-projection.json`.
- May be read/used by the provider through native skill behavior.
- Must not be treated as the provider's editable deliverable.
- Must not be the store of record.
- Must not be counted as user source changes in run diffs/scoring, except when
  the artifact is being audited as a projection artifact.

The provider can see these. The provider should not natively edit these as the
skill-authoring output. If a provider changes them accidentally, the runtime
should ignore/regenerate them or flag projection tampering rather than importing
them as user-authored skills.

### 2. Actual skills to preload

Actual skills to preload are FirmCLI/client-owned source assets or validated
registry entries that the brain can preload, project, or expose as fallback
context.

Examples today:

```text
skills/model_native/<skill_id>/
skills/provider_native/common/<skill_id>/
skills/provider_native/codex/<skill_id>/
skills/provider_native/claude/<skill_id>/
```

Future registry-backed examples:

```text
<client-owned-skill-registry>/<skill_id>/
```

Properties:

- Store of record for packaged FirmCLI skills and validated imported skills.
- Owned by FirmCLI/client runtime policy, not by the provider.
- Not written into global user provider folders.
- Not directly edited by provider-native file tools.
- Used by `load_skills(skill_ids=[...])` for API providers or fallback.
- Used to generate projected preloaded skills for CLI providers.
- Validated deterministically before becoming available as fallback context or
  future projection source.

The provider should not see or edit this surface directly unless FirmCLI
explicitly exposes a runtime copy for a bounded repair flow. API providers depend
on this surface because they cannot discover `.codex` or `.claude` native skill
folders.

### 3. User-owned skills

User-owned skills are skills or provider config files physically in the target
workspace that the user can see and own.

Examples:

```text
<target-workspace>/.codex/skills/<skill_id>/
<target-workspace>/.claude/skills/<skill_id>/
<target-workspace>/.agents/...
```

Properties:

- Visible to the user because they live in the target workspace.
- Visible to CLI providers through normal native file/workspace access.
- Natively editable by the provider when the user asks for skill authoring or
  the Wave 1.6-B skill creator/sync mode is active.
- Not automatically trusted.
- Not the canonical FirmCLI product store until synced and validated.
- Must be watched by the brain when the product is in native skill sync mode.
- Must sync into an actual skill source/registry before API fallback can be
  considered successful.

The provider can see these and, in the correct mode, may edit these natively.

## Provider Visibility And Edit Rule

| Surface | Provider can see? | Provider can natively edit? | Store of record? |
|---|---:|---:|---:|
| Projected preloaded skills | Yes | No | No |
| Actual skills to preload | No by default | No | Yes |
| User-owned skills | Yes | Yes, in skill-authoring/sync mode | No until validated/synced |

The exception for actual skills to preload is a bounded FirmCLI runtime-copy
repair flow. Even there, the provider repairs a copy that the brain validates;
it does not mutate installed source packages directly.

## Link To Wave 1.6-B Actions

The Wave 1.6-B actions spec owns the future behavior where a provider natively
authors or repairs user-owned skills, then asks FirmCLI to synchronize them.

The intended loop is:

1. Provider edits user-owned `.codex/skills` or `.claude/skills` files natively.
2. Provider returns `sync_skills` or reaches an automatic scan point.
3. FirmCLI scans changed user-owned skill folders.
4. FirmCLI validates layout, manifests, paths, safety boundaries, and fallback
   renderability.
5. Valid skills are copied into the client-owned actual-skill registry.
6. API providers can then use those synced skills through `load_skills` or the
   registry-backed equivalent.
7. Future CLI-provider runs may project those validated skills back into
   provider-native runtime views.

This is why the sync requirement matters: without syncing user-owned native skill
folders into the client-owned actual-skill registry, API fallback cannot be
considered successful.

## Workspace Diff Rule

Run diffs and benchmark scoring must not blanket-ignore all `.codex`, `.claude`,
or `.agents` directories.

Correct rule:

- Ignore marker-proven FirmCLI-generated projected preloaded skill subtrees.
- Do not ignore user-owned native skill changes merely because the path starts
  with `.codex`, `.claude`, or `.agents`.
- Record generated projection hashes separately from user-authored skill changes.
- If a path is ambiguous, prefer showing it in artifacts with provenance rather
  than hiding it.

The current code still has a broad path-based exclusion. Wave 1.6-B must narrow
that filter when native skill authoring/sync is implemented.

## Documentation Wording Rules

Use these terms consistently:

- "actual skills to preload" for the source packages or validated registry that
  FirmCLI owns and can load/project.
- "projected preloaded skills" for generated provider-visible runtime views.
- "user-owned skills" for target-workspace native provider skill folders the user
  can see and the provider may edit natively in skill-authoring mode.
- "client-owned registry" for the internal validated store that bridges user
  native authoring to API fallback and future projection.

Avoid these ambiguous phrases unless immediately qualified:

- "client-owned `.codex`" - a `.codex` folder inside the target workspace is
  user-owned by definition.
- "runtime `.codex` is user config" - projected preloaded skills are generated
  runtime views, not the user's provider config.
- "provider-native skills are editable" - only user-owned native skill folders
  are natively editable; projected preloads are read/use surfaces.

## Current Implementation Boundary

Implemented today:

- FirmCLI-owned actual provider-native skill source packages exist under
  `skills/provider_native/`.
- CLI providers receive projected preloaded skills in run-local `.codex/skills`
  or `.claude/skills` views.
- API providers use deterministic `load_skills` fallback.
- The projection layer refuses to overwrite unmarked existing native skill
  directories.

Not implemented today:

- User-owned `.codex` / `.claude` skill folder import.
- `sync_skills`.
- `skill_ok`.
- Skill creator mode.
- Client-owned registry populated from user-owned native skill folders.
- Registry-backed API fallback for provider-created skills.
- Marker/provenance-based narrowing of the workspace diff exclusion.
